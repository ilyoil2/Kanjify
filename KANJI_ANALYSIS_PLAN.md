# 한자 계층 분석 시스템(Kanjify Analysis) 구축 계획

## 1. 개요
한자 또는 히라가나 단어 입력 시, AI를 활용하여 해당 단어를 구성하는 한자의 이름, 재귀적 부수 구조, 그리고 유래를 분석하여 제공하는 서비스.

## 2. 핵심 기능 요구사항
- **계층적 파자(破字) 분석**: 한자를 기본 부수 단위까지 재귀적으로 분해하여 구조를 시각화.
- **AI 추론 기반**: 정적 DB가 아닌 LLM의 지식을 활용한 유연한 분석.
- **엄격한 신뢰도 제어**: 환각(Hallucination) 방지를 위한 강한 제약 조건 적용.
- **2단계 검증 프로세스**: AI 응답 후 서버 측에서 데이터 형식 및 유효성 검사.

## 3. AI 프롬프트 전략 (5단계 엄격 규칙)
AI가 답변을 생성할 때 반드시 준수해야 할 페르소나와 제약 조건입니다.

1. **불확실성 배제 (Strict "I don't know")**
   - 정보가 불확실하거나 확신이 없는 경우 절대 추측하지 않음.
   - 근거가 부족할 경우 해당 항목에 "정보 없음"을 명시적으로 출력.
2. **실제 한자 구성요소 기반 분해 (Standard Components Only)**
   - 자의적인 의미 분할 금지.
   - 반드시 실제 존재하는 한자 구성요소(부수 포함) 단위로만 분해.
3. **재귀적 분해 및 중단 규칙 (Recursive Decomposition Limits)**
   - 구성요소는 가능한 경우 끝까지 재귀적으로 분해하되, **의미 없는 최소 단위(획 단위) 분해는 엄격히 금지**함.
   - 구성요소는 일반적으로 **독립적인 한자 형태로 인식 가능한 단위**까지만 분해함.
4. **자기참조 및 순환참조 금지 (No Self-Reference)**
   - 분해된 결과물에 자기 자신이 포함될 수 없음 (예: 間 → 門 + 間 불가).
5. **출력 형식 고정 (Fixed Format)**
   - 지정된 마크다운 형식을 엄격히 준수.

## 4. 데이터 출력 예시 (Few-shot Examples)
AI가 참고할 실제 분해 예시입니다.

**예시 1 (기본 구조):**
- 明 → 日 + 月
- 日 → (더 분해 불가)
- 月 → (더 분해 불가)

**예시 2 (변형 부수 포함):**
- 休 → 亻 + 木
- 亻 → 人
- 人 → (더 분해 불가)
- 木 → (더 분해 불가)

**예시 3 (동일 자소 반복):**
- 森 → 木 + 木 + 木
- 木 → (더 분해 불가)

## 5. 데이터 출력 형식 (Response Format)
```markdown
> 한자 이름
- (한자1) (뜻/훈음)
- (한자2) (뜻/훈음)

> 구성자(부수) 정보
(한자1) → (구성요소A) + (구성요소B)
(구성요소A) → (기본구성요소1) + (기본구성요소2)
(한자2) → (구성요소C) + (구성요소D)

> 단어 유래
(한자1)은 ...의 결합으로 ...한 의미를 지님.
(한자2)는 ...에서 유래하여 ...을 뜻함.
```

## 5. 검증 시스템 (2-Step Validation)
AI의 응답을 클라이언트에 전달하기 전 서버(Backend)에서 수행하는 검증 로직입니다.

- **1단계: 형식 검사 (Schema Validation)**
  - 응답이 `> 한자 이름`, `> 구성자(부수) 정보`, `> 단어 유래` 섹션을 모두 포함하는지 정규표현식으로 검증.
- **2단계: 데이터 유효성 검사 (Data Integrity)**
  - `kanji` 컬럼 또는 텍스트 내의 한자가 실제 유니코드 한자 범위 내에 있는지 확인.
  - 재귀 구조에서 무한 루프가 발생하거나 비정상적인 기호가 포함되었는지 필터링.

## 6. 향후 구현 단계
1. **Prompt Engineering**: 위 규칙을 적용한 시스템 프롬프트 작성 및 테스트.
2. **API Endpoint 구축**: AI API 호출 및 서버 측 검증 로직 구현.
3. **Frontend 연동**: 분석 결과를 계층적으로 보여주는 UI 컴포넌트 개발.
